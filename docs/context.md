## 핵심 기획 의도

대형 광물의 문제는 단순히 HP가 높고 오래 걸린다는 점이 아니다. 더 중요한 문제는 직접 이동하며 채굴하던 플레이어가 대형 광물 앞에서 갑자기 멈춰 서고, 동적인 플레이가 정적인 대기 상태로 바뀐다는 점이다.

작은 광물은 이동, 접촉, 파괴, 보상이 빠르게 이어져 리듬감이 좋다. 반면 큰 광물은 오래 때려야 하고, 플레이어가 할 수 있는 선택이 줄어들어 답답하게 느껴질 수 있다.

브레이크 시스템은 이 문제를 해결하기 위한 장치다. 대형 광물을 단순히 오래 때리는 대상이 아니라, 약점을 노출시키고 공략하면 일정 시간 동안 파괴 속도가 크게 빨라지는 대상으로 바꾼다.

핵심 메시지:

- 기다림 자체가 문제가 아니라, 플레이어에게 약속한 개입 방식이 끊기는 것이 문제다.
- 직접 이동 채굴 게임에서는 플레이어가 계속 위치를 잡고 움직일 이유가 있어야 한다.
- 브레이크 시스템은 대형 광물 앞에서도 플레이어를 관전자가 아니라 공략하는 사람으로 유지시키는 장치다.
- 약점은 단순 시각 요소가 아니라 움직일 이유를 만드는 공간이다.
- 브레이크 타임은 컨트롤에 대한 보상이며, 일정한 채굴 리듬에 속도 변화를 만드는 장치다.

포트폴리오용 핵심 문장:

> 대형 광물 구간의 문제는 긴 파괴 시간 자체가 아니라, 직접 이동하며 채굴하던 플레이어가 갑자기 기다리는 관전자로 바뀐다는 점이라고 판단했다. 브레이크 시스템은 대형 광물 앞에서도 플레이어가 위치를 잡고 약점을 공략하는 역할을 유지하도록 설계한 장치다.

## 현재 구현 방향

게임은 탑뷰 시점의 시스템 검증용 프로토타입이다.

현재 탑뷰는 원작 시점과 완전히 같지는 않지만, 브레이크 시스템의 조작 리듬, 약점 공략, 카고 루프를 빠르게 검증하기 위한 V1 버전으로 본다.

향후 V2 방향:

- Spin Miner 원작 톤에 가까운 고정 방향/2.5D 느낌
- 광산을 파고 들어가는 진행감
- 모바일 세로 화면에 맞춘 UI
- 더 실제 상용 게임에 가까운 카메라/공간 연출

## 현재 주요 기능

- HTML5 Canvas 단일 파일 게임
- WASD 이동
- 마우스 클릭/드래그 이동
- 플레이어 주변 6개의 곡괭이 자동 회전
- 플레이어가 광물을 덮고 지나가지 못하는 충돌 이동
- 광물 내부 숫자 표시
- 숫자는 `12.34A`처럼 소수점 2자리 표시
- HP바 제거
- 광물 HP 감소는 숫자 감소와 어두워지는 오버레이로 인지
- 광물 간 스폰 간격 강화
- 겹칠 자리가 없으면 해당 광물 생성을 건너뜀
- 챕터마다 소형/중형/대형/초대형 광물 구성
- 소형 광물 수를 줄이고 중형/대형 비중을 올림
- 챕터가 올라갈수록 대형 광물 크기와 HP 증가
- 기존 대형 광물 HP 2배
- 초대형 광물 1개 추가 시도
- 초대형 광물은 기존 대형 기준 HP 3배
- 대형/초대형 광물은 약 2초간 공격해야 약점 노출
- 약점 공격 시 균열 게이지 증가
- 브레이크 게이지 증가 속도는 기존보다 2배
- 약점이 아닌 부위도 HP 감소
- 약점 공격이 더 효율적
- 약점은 일정 시간마다 위치 이동
- 브레이크 상태는 영구 지속이 아니라 약 5초 제한 시간
- 브레이크 중 몸통 대미지 증가
- 브레이크 중 약점 표시 숨김
- 브레이크 남은 시간은 게이지로 표시
- 브레이크 종료 시 균열 0%, 약점 숨김 상태로 복귀
- 광물을 캐면 플레이어 개인 카고에 적재
- 플레이어가 BASE에 닿아야 스테이지 카고에 반영
- 개인 카고가 가득 차면 추가 채굴 제한
- 개인 카고가 가득 차면 말풍선 표시: "베이스로 가 카고를 비우세요!"
- 모든 광물을 제거하고 개인 카고를 비우면 챕터 클리어

## 작업할 때 주의점

- 사용자는 비개발자이므로 설명은 기획자 관점으로 쉽게 해야 한다.
- 코드를 수정할 때는 구현 설명보다 플레이 감각 변화와 기획 의도 연결이 중요하다.
- 포트폴리오 톤은 겸손하고 분석적이어야 한다.
- Supercent 공식 프로젝트처럼 보이면 안 된다.
- 원작을 비판하기보다, 채용 포지션 관점에서 개선 가설을 제안하는 톤이 좋다.
- 기능을 추가할 때마다 README 또는 reference 문서와 실제 게임 설명이 어긋나지 않게 맞춘다.
- 큰 변경 후에는 JavaScript 문법 검사를 한다.
- GitHub에 올리기 전에는 사용자가 "올려"라고 명시했는지 확인한다.
